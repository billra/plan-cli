Plan.py Test Suite
==================

Initial State & The Blank Slate
-------------------------------
The file starts empty.
    >>> plan_file.exists()
    False

The application enforces that the root must begin at 1.
    >>> cmd("2", "Invalid Root")
    ValidationError: blank slate must start at 1

    >>> cmd("insert", "1", "First Root")
    >>> cmd()
    ☐ 1 "First Root"

Deleting the only task should safely return us to an empty state.
    >>> cmd("delete", "1")
    >>> cmd()

Atomic Addition & Order Independence
------------------------------------
Because `add_or_replace` sorts pairs before processing, we should be able to
define a valid tree in a completely random order, and it will construct properly.
    >>> cmd("1.1.1", "Leaf", "1", "Root", "1.1", "Branch")
    >>> cmd()
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☐ 1.1.1 "Leaf"

Adding a task missing its immediate parent fails validation.
    >>> cmd("1.2.1", "Orphan")
    ValidationError: parent 1.2 does not exist

Adding a task that skips a sibling index fails validation.
    >>> cmd("1.3", "Skipper")
    ValidationError: sibling gap - expected 1.2

If a batch contains an invalid operation, the entire command must abort atomically (no partial saves).
    >>> cmd("1.2", "Valid", "1.4", "Invalid Gap")
    ValidationError: sibling gap - expected 1.3
    >>> cmd()
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☐ 1.1.1 "Leaf"

Idempotency & Replacement State Resets
--------------------------------------
Let's complete the entire tree.
    >>> cmd("complete", "1")
    >>> cmd("1")
    ☒ 1 "Root"
    ☒ 1.1 "Branch"
    ☒ 1.1.1 "Leaf"

Completing an already complete task should be idempotent (no-op, no crash).
    >>> cmd("complete", "1.1.1")
    >>> cmd("1.1.1")
    ☒ 1.1.1 "Leaf"

Replacing an existing task updates its description AND forces its state to incomplete.
Because of propagation, this must also bubble up and incomplete its ancestors.
    >>> cmd("1.1.1", "Updated Leaf")
    >>> cmd("1")
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☐ 1.1.1 "Updated Leaf"

Adding a new child to a completed parent must also incomplete the parent!
Let's complete it all again, then add 1.1.2.
    >>> cmd("complete", "1")
    >>> cmd("1.1.2", "New Sibling")
    >>> cmd("1")
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☒ 1.1.1 "Updated Leaf"
    ☐ 1.1.2 "New Sibling"

Deep Shifting Constraints (Insert)
----------------------------------
Inserting at the root pushes EVERYTHING down, and changes all prefixes.
    >>> cmd("insert", "1", "New Root")
    >>> cmd()
    ☐ 1 "New Root"
    ☐ 2 "Root"
    ☐ 2.1 "Branch"
    ☒ 2.1.1 "Updated Leaf"
    ☐ 2.1.2 "New Sibling"

Inserting into a deep node shifts its siblings right (+1) and updates their children.
Let's insert at 2.1.1. The old 2.1.1 and 2.1.2 should become 2.1.2 and 2.1.3.
    >>> cmd("insert", "2.1.1", "Inserted Leaf")
    >>> cmd("2.1")
    ☐ 2.1 "Branch"
    ☐ 2.1.1 "Inserted Leaf"
    ☒ 2.1.2 "Updated Leaf"
    ☐ 2.1.3 "New Sibling"

Deep Shifting Constraints (Delete)
----------------------------------
Deleting a middle node pulls subsequent siblings left (-1).
Deleting 2.1.2 should shift 2.1.3 down to 2.1.2.
    >>> cmd("delete", "2.1.2")
    >>> cmd("2.1")
    ☐ 2.1 "Branch"
    ☐ 2.1.1 "Inserted Leaf"
    ☐ 2.1.2 "New Sibling"

Deleting a parent wipes out its children entirely.
    >>> cmd("delete", "2.1")
    >>> cmd("2")
    ☐ 2 "Root"

State Propagation (Cascading & Bubbling)
----------------------------------------
Marking a parent incomplete must aggressively force all descendants to incomplete.
Let's build out a small completed tree first.
    >>> cmd("2.1", "A", "2.1.1", "B", "2.1.2", "C")
    >>> cmd("complete", "2")
    >>> cmd("2")
    ☒ 2 "Root"
    ☒ 2.1 "A"
    ☒ 2.1.1 "B"
    ☒ 2.1.2 "C"

Now incomplete the top level node. Everything underneath must become incomplete.
    >>> cmd("incomplete", "2")
    >>> cmd("2")
    ☐ 2 "Root"
    ☐ 2.1 "A"
    ☐ 2.1.1 "B"
    ☐ 2.1.2 "C"

Conversely, completing the last incomplete child must bubble up and complete its ancestors.
First, completing 2.1.1 does nothing to 2.1 or 2 yet.
    >>> cmd("complete", "2.1.1")
    >>> cmd("2")
    ☐ 2 "Root"
    ☐ 2.1 "A"
    ☒ 2.1.1 "B"
    ☐ 2.1.2 "C"

Now we complete 2.1.2. Because both 2.1.1 and 2.1.2 are complete, 2.1 completes.
Because 2.1 is the only child of 2, 2 also completes.
    >>> cmd("complete", "2.1.2")
    >>> cmd("2")
    ☒ 2 "Root"
    ☒ 2.1 "A"
    ☒ 2.1.1 "B"
    ☒ 2.1.2 "C"

String Escaping & Empty Descriptions
------------------------------------
Double quotes and backslashes must be safely escaped in storage and unescaped in memory.
Tasks can also have completely empty descriptions.
    >>> cmd("3", 'Has "quotes" and \\slashes', "3.1", "")
    >>> cmd("3")
    ☐ 3 "Has \"quotes\" and \\slashes"
    ☐ 3.1 ""

Invalid Number Parsing
----------------------
The `parse_num` regex requires strict dot-separated integers.
    >>> cmd("delete", "1.")
    ValidationError: invalid task number format: 1.

    >>> cmd("delete", ".1")
    ValidationError: invalid task number format: .1

    >>> cmd("delete", "1..1")
    ValidationError: invalid task number format: 1..1

    >>> cmd("delete", "A.1")
    ValidationError: invalid task number format: A.1

Target Missing Errors
---------------------
Attempting to operate on non-existent targets safely throws validation errors.
    >>> cmd("complete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("incomplete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("delete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("9.9")
    ValidationError: task 9.9 does not exist

CLI Usage & Argument Validation
-------------------------------
Providing unrecognized commands, or the wrong number of arguments for strict commands, raises a UsageError.
    >>> cmd("dance")
    UsageError: unrecognized command or invalid arguments

    >>> cmd("insert", "1")
    UsageError: unrecognized command or invalid arguments

    >>> cmd("1", "Desc", "2")
    UsageError: unrecognized command or invalid arguments

Strict Number Formatting (No Leading Zeros)
-------------------------------------------
The numbering strictly prohibits leading zeros at any level of the hierarchy to prevent ambiguity.
    >>> cmd("01", "Leading Zero")
    ValidationError: invalid task number format: 01

    >>> cmd("1.02", "Inner Leading Zero")
    ValidationError: invalid task number format: 1.02

File Parsing Resilience (Empty Lines & Corruption)
--------------------------------------------------
The tool safely ignores empty lines in the storage file. We can test this by writing directly to the underlying file.
    >>> _ = plan_file.write_text("\n\n☐ 1 \"Surviving Task\"\n\n\n", encoding="utf-8")
    >>> cmd()
    ☐ 1 "Surviving Task"

If a user manually corrupts the file with malformed text, the tool completely aborts. It throws an I/O error to prevent the application from overwriting and destroying the user's data.
    >>> _ = plan_file.write_text("☐ 1 Missing Quotes Around Description\n", encoding="utf-8")
    >>> cmd()
    OSError: malformed line 1 in plan.txt

Let's delete the corrupted file so the application resets to a blank slate for the final test.
    >>> plan_file.unlink()
