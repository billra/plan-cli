Plan.py Test Suite
==================

Initial State
-------------
    >>> plan_file.exists()
    False

1. The Atomic Constraint
------------------------
The `plan <n> "<d>" ...` command must validate ALL pairs before writing. If there is a
gap or error at the end of the input, nothing should be written to the file.
    >>> cmd("1", "Root", "1.1", "Child", "1.3", "Gap Child")
    ValidationError: sibling gap - expected 1.2
    >>> plan_file.exists()
    False

2. Building a Complex Tree
--------------------------
Let's build a multi-root, multi-depth tree to test complex chain reactions.
    >>> cmd("1", "Project Alpha", "1.1", "Phase 1", "1.1.1", "Task A", "1.1.2", "Task B")
    >>> cmd("1.2", "Phase 2", "2", "Project Beta", "2.1", "Phase 1")
    >>> cmd()
    ☐ 1 "Project Alpha"
    ☐ 1.1 "Phase 1"
    ☐ 1.1.1 "Task A"
    ☐ 1.1.2 "Task B"
    ☐ 1.2 "Phase 2"
    ☐ 2 "Project Beta"
    ☐ 2.1 "Phase 1"

3. Chain Reaction Completeness (Upward)
---------------------------------------
Completing one child of a multi-child parent shouldn't bubble up.
    >>> cmd("complete", "1.1.1")
    >>> cmd("1.1")
    ☐ 1.1 "Phase 1"
    ☒ 1.1.1 "Task A"
    ☐ 1.1.2 "Task B"

Completing the final sibling should trigger a chain reaction, completing
the parent (1.1). However, Root (1) should remain incomplete because 1.2 is incomplete.
    >>> cmd("complete", "1.1.2")
    >>> cmd("1")
    ☐ 1 "Project Alpha"
    ☒ 1.1 "Phase 1"
    ☒ 1.1.1 "Task A"
    ☒ 1.1.2 "Task B"
    ☐ 1.2 "Phase 2"

Completing the last top-level dependency (1.2) should now complete the Root (1).
    >>> cmd("complete", "1.2")
    >>> cmd("1")
    ☒ 1 "Project Alpha"
    ☒ 1.1 "Phase 1"
    ☒ 1.1.1 "Task A"
    ☒ 1.1.2 "Task B"
    ☒ 1.2 "Phase 2"

4. Deep Subtree Shifting (Insert)
---------------------------------
Inserting a task in the middle of the tree (1.1) must shift the previous 1.1 to 1.2.
Crucially, ALL children of the shifted parent must have their prefixes updated
(e.g., old 1.1.1 becomes 1.2.1). Root 2 should remain completely unaffected.
    >>> cmd("insert", "1.1", "New Phase 0")
    >>> cmd()
    ☒ 1 "Project Alpha"
    ☐ 1.1 "New Phase 0"
    ☒ 1.2 "Phase 1"
    ☒ 1.2.1 "Task A"
    ☒ 1.2.2 "Task B"
    ☒ 1.3 "Phase 2"
    ☐ 2 "Project Beta"
    ☐ 2.1 "Phase 1"

Notice that inserting the incomplete "New Phase 0" at 1.1 caused the root (1)
to immediately bubble to incomplete (☐). The implementation handled the bubble!

5. Replace and Upward Interruption
----------------------------------
Let's complete the whole tree first.
    >>> cmd("complete", "1")
    >>> cmd("complete", "2")

Replacing a deep task resets it to incomplete. This should interrupt the complete
state of its parent and its root, without affecting its siblings.
    >>> cmd("1.2.2", "Revised Task B")
    >>> cmd("1")
    ☐ 1 "Project Alpha"
    ☒ 1.1 "New Phase 0"
    ☐ 1.2 "Phase 1"
    ☒ 1.2.1 "Task A"
    ☐ 1.2.2 "Revised Task B"
    ☒ 1.3 "Phase 2"

6. Incomplete Cascading (Downward)
----------------------------------
Marking a top-level parent incomplete should force all its descendants incomplete.
    >>> cmd("incomplete", "2")
    >>> cmd("2")
    ☐ 2 "Project Beta"
    ☐ 2.1 "Phase 1"

7. Deep Subtree Shifting (Delete)
---------------------------------
Deleting a middle sibling that has children should completely erase that subtree,
and properly shift the subsequent siblings back down to close the gap.
If we delete 1.2 (Phase 1 and its children), 1.3 should become the new 1.2.
    >>> cmd("delete", "1.2")
    >>> cmd()
    ☐ 1 "Project Alpha"
    ☒ 1.1 "New Phase 0"
    ☒ 1.2 "Phase 2"
    ☐ 2 "Project Beta"
    ☐ 2.1 "Phase 1"

8. Edge Boundary Validations
----------------------------
Deleting a non-existent child inside a valid parent:
    >>> cmd("delete", "1.3")
    ValidationError: task 1.3 does not exist

Inserting at a deeper boundary gap:
    >>> cmd("insert", "1.1.2", "Gap")
    ValidationError: sibling gap - expected 1.1.1
