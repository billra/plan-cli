Plan.py Test Suite
==================

Initial State & The Blank Slate
-------------------------------
The file starts empty.
    >>> plan_file.exists()
    False

The application enforces that the root must begin at 1.
    >>> cmd("2", "Invalid Root")
    ValidationError: blank slate must start at 1

    >>> cmd("insert", "1", "First Root")
    >>> cmd()
    ☐ 1 "First Root"

Deleting the only task should safely return us to an empty state.
    >>> cmd("delete", "1")
    >>> cmd()

Atomic Addition & Order Independence
------------------------------------
Because `add_or_replace` sorts pairs before saving, we should be able to
define a valid tree in a completely random order, and it will construct properly.
    >>> cmd("1.1.1", "Leaf", "1", "Root", "1.1", "Branch")
    >>> cmd()
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☐ 1.1.1 "Leaf"

Adding a task missing its immediate parent fails validation.
    >>> cmd("1.2.1", "Orphan")
    ValidationError: parent 1.2 does not exist

Adding a task that skips a sibling index fails validation.
    >>> cmd("1.3", "Skipper")
    ValidationError: sibling gap - expected 1.2

Idempotency & Replacement State Resets
--------------------------------------
Let's complete the entire tree.
    >>> cmd("complete", "1")
    >>> cmd("1")
    ☒ 1 "Root"
    ☒ 1.1 "Branch"
    ☒ 1.1.1 "Leaf"

Completing an already complete task should be idempotent (no-op, no crash).
    >>> cmd("complete", "1.1.1")
    >>> cmd("1.1.1")
    ☒ 1.1.1 "Leaf"

Replacing an existing task updates its description AND forces its state to incomplete.
Because of propagation, this must also bubble up and incomplete its ancestors.
    >>> cmd("1.1.1", "Updated Leaf")
    >>> cmd("1")
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☐ 1.1.1 "Updated Leaf"

Adding a new child to a completed parent must also incomplete the parent!
Let's complete it all again, then add 1.1.2.
    >>> cmd("complete", "1")
    >>> cmd("1.1.2", "New Sibling")
    >>> cmd("1")
    ☐ 1 "Root"
    ☐ 1.1 "Branch"
    ☒ 1.1.1 "Updated Leaf"
    ☐ 1.1.2 "New Sibling"

Deep Shifting Constraints (Insert)
----------------------------------
Inserting at the root pushes EVERYTHING down, and changes all prefixes.
    >>> cmd("insert", "1", "New Root")
    >>> cmd()
    ☐ 1 "New Root"
    ☐ 2 "Root"
    ☐ 2.1 "Branch"
    ☒ 2.1.1 "Updated Leaf"
    ☐ 2.1.2 "New Sibling"

Inserting into a deep node shifts its siblings right (+1) and updates their children.
Let's insert at 2.1.1. The old 2.1.1 and 2.1.2 should become 2.1.2 and 2.1.3.
    >>> cmd("insert", "2.1.1", "Inserted Leaf")
    >>> cmd("2.1")
    ☐ 2.1 "Branch"
    ☐ 2.1.1 "Inserted Leaf"
    ☒ 2.1.2 "Updated Leaf"
    ☐ 2.1.3 "New Sibling"

Deep Shifting Constraints (Delete)
----------------------------------
Deleting a middle node pulls subsequent siblings left (-1).
Deleting 2.1.2 should shift 2.1.3 down to 2.1.2.
    >>> cmd("delete", "2.1.2")
    >>> cmd("2.1")
    ☐ 2.1 "Branch"
    ☐ 2.1.1 "Inserted Leaf"
    ☐ 2.1.2 "New Sibling"

Deleting a parent wipes out its children entirely.
    >>> cmd("delete", "2.1")
    >>> cmd("2")
    ☐ 2 "Root"

Downward Cascading Constraints
------------------------------
Marking a parent incomplete must aggressively force all descendants to incomplete.
Let's build out a small completed tree first.
    >>> cmd("2.1", "A", "2.1.1", "B", "2.1.2", "C")
    >>> cmd("complete", "2")
    >>> cmd("2")
    ☒ 2 "Root"
    ☒ 2.1 "A"
    ☒ 2.1.1 "B"
    ☒ 2.1.2 "C"

Now incomplete the top level node. Everything underneath must become incomplete.
    >>> cmd("incomplete", "2")
    >>> cmd("2")
    ☐ 2 "Root"
    ☐ 2.1 "A"
    ☐ 2.1.1 "B"
    ☐ 2.1.2 "C"

Invalid Number Parsing
----------------------
The `parse_num` regex requires strict dot-separated integers.
    >>> cmd("delete", "1.")
    ValidationError: invalid task number format: 1.

    >>> cmd("delete", ".1")
    ValidationError: invalid task number format: .1

    >>> cmd("delete", "1..1")
    ValidationError: invalid task number format: 1..1

    >>> cmd("delete", "A.1")
    ValidationError: invalid task number format: A.1

Target Missing Errors
---------------------
Attempting to operate on non-existent targets safely throws validation errors.
    >>> cmd("complete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("incomplete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("delete", "9.9")
    ValidationError: task 9.9 does not exist

    >>> cmd("9.9")
    ValidationError: task 9.9 does not exist
